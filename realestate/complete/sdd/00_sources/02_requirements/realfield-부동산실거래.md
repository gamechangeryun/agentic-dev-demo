## 1. 사업 개요

### 1.1 배경

부동산 실거래가는 국토교통부가 신고 접수를 통해 수집하고 공공데이터포털(data.go.kr)을 통해 공개한다. 그러나 데이터는 시군구·계약월 단위로 분절되어 제공되고, 거래금액이 만원 단위 콤마 문자열로 내려오며, 계약 해제(거래 취소) 건이 혼재한다. 분석 수요자는 매월 250개 시군구를 반복 호출하고, 표준 스키마로 정규화하며, 해제 거래를 집계에서 제외하는 작업을 각자 수행해야 한다. 본 사업은 이 수집·정규화·집계 부담을 플랫폼이 일괄 흡수해, 신뢰 가능한 표준 실거래 원장과 시세 통계를 제공한다.

### 1.2 목적

1. 전국 시군구의 아파트 매매 실거래를 매월 자동 수집해 단일 표준 스키마로 통합 적재한다.
2. 외부 공개 API의 지연·장애·트래픽 한도에도 수집 파이프라인이 멈추지 않게 한다.
3. 거래금액 정규화와 해제 거래 처리를 플랫폼이 일관되게 강제해, 수요자별 집계 편차를 제거한다.
4. 동일 구간 재수집에도 중복이 생기지 않는 멱등 적재를 보장한다.
5. 시세 통계를 거래원장과 분리된 조회 모델에서 저지연으로 제공한다.

### 1.3 기대 효과

- 수요자별 중복 수집·정규화 제거로 데이터 품질 편차 해소.
- 외부 장애가 분석 서비스로 전파되지 않는 운영 안정성 확보.
- 표준 원장·표준 통계 기반의 의사결정 지원.

## 2. 사업 범위

### 2.1 포함 범위 (In-Scope)

- 국토교통부 아파트 매매 실거래가 OpenAPI(data.go.kr 15126469/15126468) 연계 수집.
- 전국 250개 시군구 × 월 단위 수집 배치.
- 표준 거래 스키마(AptTransaction) 정규화·멱등 적재.
- 거래 조회 API, 시세 통계(중위 거래금액·중위 ㎡당 단가) 조회 API.
- 계약 해제(취소) 거래의 식별·집계 제외 처리.

### 2.2 제외 범위 (Out-of-Scope)

- 아파트 외 부동산 유형(연립·다세대·오피스텔·단독/다가구) 수집. (확장 후보로만 명시)
- 전월세 실거래(data.go.kr 15126474) 수집. (2단계 확장 후보)
- 실거래가 예측·AVM 등 분석 모델링.
- 대국민 포털 UI. 본 사업은 수집·적재·조회 백엔드와 게이트웨이까지를 범위로 한다.

### 2.3 규모

| 항목 | 규모 |
| --- | --- |
| 대상 지역 | 전국 250개 시군구 |
| 수집 주기 | 매월(계약월 단위), 과거 구간 백필 포함 |
| 누적 데이터 | 수억 건 규모 |
| 서비스 수 | 마이크로서비스 20+ (코어: 수집/거래원장/분석/플랫폼) |

### 2.4 기술 스택(전제)

Java 21, Spring Boot 3.5.x, Spring Cloud 2025.0(Northfields), Gradle 멀티모듈, resilience4j(회복력), Spring Cloud Gateway(게이트웨이), Eureka(디스커버리), Config Server(설정 외부화).

## 8. 제약사항 (CONR)

| 번호 | 내용 |
| --- | --- |
| CONR-001 | 외부 API는 일일 트래픽 한도(개발 10,000건)와 운영시간 제약이 있다. 본 시스템 설계는 이 한도 안에서 동작해야 한다. |
| CONR-002 | 외부 응답은 XML이며 모든 값이 문자열로 전달된다. 숫자·날짜는 시스템이 형 변환 책임을 진다. |
| CONR-003 | 거래금액은 만원 단위이며 천 단위 콤마·선행 공백을 포함한다. 원 단위 변환은 시스템 책임이다. |
| CONR-004 | 동(棟)·등기일자는 소유권 이전등기 완료 건에 한해서만 채워진다. 미완료 건은 공백이며 결측을 허용해야 한다. |
| CONR-005 | 계약 해제 건은 원천에서 사후에 표기될 수 있다(cdealType). 재수집 시 해제 상태 변경을 반영해야 한다. |
| CONR-006 | 스택은 Java 21 / Spring Boot 3.5.x / Spring Cloud 2025.0 / Gradle 멀티모듈로 고정한다. |
| CONR-007 | 외부 라이브러리·CLI 신규 도입은 최소화하며, 회복력은 resilience4j로 한정한다. |

## 10. 핵심 요구 (정제 전 원문 발췌, 보존)

1. 시군구·계약월 단위로 실거래를 수집해 표준 스키마로 적재한다.
2. 외부 API 지연·실패에도 파이프라인이 멈추지 않게 한다(회복력).
3. 거래금액 콤마 문자열을 정확히 변환하고 해제거래를 집계에서 제외한다(정합).
4. 같은 구간을 재수집해도 중복이 생기지 않게 한다(멱등).
5. 시세 통계는 거래원장과 분리된 조회 모델에서 빠르게 제공한다(CQRS).
