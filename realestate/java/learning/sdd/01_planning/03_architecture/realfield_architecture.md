# 아키텍처 설계 — RealField MSA 경계 (사람의 판단 + 기각 대안)

> 출처(00_sources): 요구사항정의서·API 공개명세·데이터 명세서.
> `01_feature`의 AC-1~AC-5를 만족하는 서비스 경계를 설계하고, **사람이 판단한 결정**과 **기각한 대안**을 함께 남긴다. (SDD 2단계 산출물)

## 1. 먼저 토론한 것 (서비스 특성)
경계를 가르기 전에 세 질문에 답한다.

1. **읽기와 쓰기 부하가 다른가?** — 쓰기는 매월 250 시군구 × 계약월의 대량 배치 적재(누적 수억 건). 읽기는 시세 통계 저지연 조회(P99 ≤ 300ms, 가용성 99.9%). → **비대칭**. 수집 배치가 조회를 열화시키면 안 된다(PER-003).
2. **외부 API 장애를 어디에 가둘까?** — data.go.kr은 지연·트래픽 한도(일 1만)·XML 전용·사후 해제 표기 등 불확실성의 원천. 장애 폭발 반경을 **한 경계 안**에 가두어야 한다.
3. **지금 비동기가 필요한가?** — 현 요구는 멱등 배치 수집·조회다. 이벤트 스트리밍/CDC는 현 규모에서 결정성·운영 단순성을 해친다. → **아직 불필요**.

## 2. Bounded Context / 모듈 경계
| 모듈 | 책임 | 종류 |
| --- | --- | --- |
| `common` | 표준 스키마(AptTransaction)·정합 변환(DealAmountParser)·자연키·지역코드 | 공유 라이브러리 |
| `ingestion-service` | data.go.kr 수집·XML 파싱·정규화·회복력 | 도메인(쓰기 입구) |
| `transaction-service` | 거래원장 적재(멱등 upsert)·거래 조회 | 도메인(write model) |
| `analytics-service` | 시세 통계 집계·조회 | 도메인(read model / CQRS) |
| `api-gateway` | 단일 진입점, `/api/v1/*` 라우팅(:8080) | 인프라 |
| `service-discovery` | Eureka 서비스 등록·LB(:8761) | 인프라 |
| `config-server` | 설정·인증키 외부화 | 인프라 |

## 3. 경계 결정과 기각 대안 (← 사람이 판단한 지점)

### 결정 1 · analytics를 별도 read model로 분리 (CQRS)
- **근거**: 읽기/쓰기 부하 비대칭(§1.1). 대량 적재 중에도 시세 조회 P99를 지키려면 조회 모델을 분리해야 한다(AC-5, PER-001/003).
- **기각 대안**: 거래원장에서 직접 집계 조회 → 대량 배치 적재가 조회 인덱스와 경합해 P99 열화. **기각**.

### 결정 2 · 외부 의존을 ingestion-service에 격리
- **근거**: data.go.kr 호출·XML 파싱·회복력(재시도·서킷·타임아웃)을 한 경계에 모아 폭발 반경을 제한한다(AC-2, SIR-005). 인증키 노출면도 한곳으로 좁힌다.
- **기각 대안**: 각 서비스가 외부 호출을 분산하거나 게이트웨이에서 처리 → 회복력 정책·키 취급이 산재·중복. **기각**.

### 결정 3 · 동기 멱등 배치 (지금 메시지 브로커 도입 안 함)
- **근거**: 자연키 멱등 upsert로 재수집이 안전하고(AC-4), 단위 검증이 네트워크 없이 결정적이다. 현 규모에선 단순성이 이득.
- **기각 대안**: Kafka 이벤트 소싱/비동기 파이프라인 → 현 단계 과설계, 운영·정합 디버깅 비용↑. **확장 후보로 보류**(전월세·타 부동산 유형 확장 시 재검토).

### 결정 4 · common 공유 계약 (단일 소유, 역의존 금지)
- **근거**: 세 도메인이 표준 스키마·정합 변환·자연키를 공유한다. 한곳에서 강제해 수요자별 집계 편차를 제거한다.
- **규칙**: `common`은 어떤 도메인 모듈에도 의존하지 않는다(**역의존 금지**). 의존은 항상 `도메인 → common` 단방향.
- **기각 대안**: 서비스마다 스키마·금액 변환을 자체 구현 → 정합 편차·중복. **기각**.

### 결정 5 · 게이트웨이 단일 진입 + Eureka 디스커버리
- **근거**: 모든 내부 API를 게이트웨이 단일 진입점으로 노출(SIR-003)하고, 내부 호출은 디스커버리 LB(SIR-004). 라우팅 계약은 `05_api` 정본.

## 4. 런타임 토폴로지 · 의존 규칙
```
config-server ──(설정·키)──┐
service-discovery(Eureka) ─┤  등록/LB
                           ▼
client → api-gateway(:8080) ─/api/v1/ingest───────→ ingestion-service ──→ data.go.kr (외부, 회복력 격리)
                            ─/api/v1/transactions──→ transaction-service (write model)
                            ─/api/v1/market-stats──→ analytics-service ──(조회)──→ transaction-service
```
**아키텍처 게이트(불변 규칙)**: ① common 역의존 없음 ② 도메인 → common 단방향 ③ analytics → transaction(CQRS 조회) ④ 게이트웨이 3라우트 ⑤ 외부 호출은 ingestion 경계에만.
